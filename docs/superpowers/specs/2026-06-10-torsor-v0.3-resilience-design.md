# torsor-helper v0.3 "Resilience Release" — Design

**Goal:** Address the highest-value vibe-coding failure modes that torsor can *actually* reach — supply-chain risk from hallucinated dependencies, cross-file/blast-radius blindness, hidden coupling, and review fatigue — while holding every invariant: local-first, no required cloud/paid APIs, Markdown is the source of truth, the SQLite index is rebuildable, small dependency surface, deterministic + offline-testable.

**Provenance:** Distilled from deep web research into documented vibe-coding failure modes (GitClear AI Code Quality 2025; "Lost in the Middle" arXiv 2307.03172; USENIX Security '25 slopsquatting / Spracklen et al.; GitGuardian State of Secrets Sprawl 2026; CSA/Veracode security surveys; flow-debt & Truck-Factor papers). The research's verify phase was starved by a rate limit, so the qualitative failure modes (well-established, recognizable sources) are taken as sound while exact percentages are treated as "reported."

**Scope = 4 features** (the recommended core). Each maps to a researched failure mode and builds on what v0.2 already ships.

---

## Non-negotiable invariants (every task)
1. No new **required** dependency. Optional/best-effort paths degrade gracefully.
2. Markdown stays the source of truth; new persistent state is rebuildable index (`.index/`) or committed config — never deletes prose.
3. Determinism: no `datetime.now()`/RNG in ranking/search/index/guard paths. Git history and the filesystem are the only external inputs, and absence → empty result.
4. Conservative on false positives — especially the dependency check, which is **advisory**, never a hard failure by default.

---

## P1 — Slopsquatting guard (offline dependency-existence check)  *(new; flagship)*

**Failure mode:** LLMs reference non-existent packages at scale (USENIX '25: ~5.2% commercial / 21.7% OSS; 43% reproduced) — a registerable supply-chain attack ("slopsquatting"). The research explicitly names *offline import-existence validation* as the missing, locally-performable defense.

**Design — a new `deps.py` module, fully offline.** For the changed `*.py` files, extract each top-level absolute import name and flag any that resolves to **no known package**. "Known" is the union of:
- **stdlib** — `sys.stdlib_module_names` (Python ≥3.10; we require ≥3.11).
- **installed import names** — scan the project's own virtualenv (`<root>/.venv`, `venv`) `site-packages/*.dist-info/top_level.txt` (+ `*.egg-info/top_level.txt`, falling back to the normalized dist name). This is the *accurate* source: what's actually importable in the project. (torsor runs as its own tool, so we must read the project's env, not torsor's.)
- **first-party** — top-level packages/modules in the repo: dirs containing `__init__.py` at the root or under `src/`, plus top-level `*.py` stems. Covers `src/`-layout (`torsor_helper`), flat layouts, and single-file modules.
- **declared deps (fallback)** — names from `pyproject.toml` (`[project].dependencies` + `optional-dependencies`) and `requirements*.txt`, normalized (lowercase, unify `-_.`), passed through a small built-in dist→import **alias table** (`pyyaml`→`yaml`, `pillow`→`PIL`, `beautifulsoup4`→`bs4`, `scikit-learn`→`sklearn`, `opencv-python`→`cv2`, …).

Relative imports (`from . import x`) are skipped. An import flagged only when it matches **none** of the above. Output as advisory findings (severity `info`), surfaced three ways:
- `torsor deps [files]` CLI (defaults to git-changed files; `--strict` to exit non-zero).
- a `check_dependencies(files?)` MCP tool.
- a Coach `kind="phantom_dep"` recommendation in `recommend()`.

**Files:** new `deps.py`; `operations.check_dependencies` / `new_dependency_findings`; `cli.py` (`deps`), `server.py` (tool), `coach/report.py` (rec). **Risk:** med — false positives. Mitigated by: union of four sources, venv as ground truth, advisory severity, conservative "flag only if unknown everywhere," and a documented best-effort note. **Tests:** offline via a fake `.venv/.../top_level.txt` + a fixture repo; assert a hallucinated import flags and stdlib/first-party/installed/declared imports don't.

## P2 — Impact analysis ("who calls this?")  *(new; reuses v0.2 edges)*

**Failure mode:** cross-file reasoning failures — regenerating one symbol cascades (auth JWT→cookie breaking dependents; renamed API fields breaking integration tests). The agent can't see the blast radius before it edits.

**Design:** surface callers of a symbol via the `symbol_edges` / `db.who_references` shipped in v0.2. `operations.impact(store, config, symbol)`:
- resolve the symbol's defining module(s) via `db.search_symbols` (exact-name match → its `module`, normalized to dotted via `cartographer._norm_module`);
- query `db.who_references(conn, dotted_module, base_name)` → `[(caller, module)]`;
- return a structured blast-radius (callers grouped by module + a total count).

Surfaced as `torsor impact <symbol>` CLI + an `impact(symbol)` MCP tool. Read-only over the existing index. **Files:** `operations.py`, `cli.py`, `server.py`. **Risk:** low.

## P3 — Temporal-coupling edges (git co-change)  *(planned fast-follow)*

**Failure mode:** hidden dependencies the import graph can't see — files that always change together but aren't linked — drive lost architectural intent and degrade handoff (Truck-Factor decay).

**Design — `coach/coupling.py`, git-gated like hotspots:**
- parse `git log --name-only --pretty=format:%H`, group `*.py` files per commit; **ignore commits touching > `max_files` (default 40)** (merges/sweeps) and files with `< min_commits` (default 3);
- for each unordered pair, `co = #commits touching both`; degree `= co / min(changes(a), changes(b))`; keep pairs with `degree ≥ threshold` (default 0.6) and `co ≥ min_commits`;
- emit a Coach `kind="coupling"` rec for the top pairs **where neither file imports the other** (checked via `symbol_edges`/`module_edges`) **and** there's no wiki link — "X and Y change together {co}× but nothing links them; document the coupling." Gated behind `conn`-present and a `git rev-parse` check (empty otherwise).

**Files:** new `coach/coupling.py`, `coach/report.py`. **Risk:** low. **Tests:** the `git_project` fixture extended with two files committed together; assert the pair surfaces and an unrelated file doesn't; non-git → empty.

## P6 — Complexity-trend "new findings only" coach view  *(planned fast-follow)*

**Failure mode:** review fatigue / decision fatigue — alerting on absolute badness nags; 35–91% of static-analysis alerts are non-actionable. Alerting on *regression since baseline* is the never-nag win.

**Design — `coach/trend.py` + a snapshot in the rebuildable index:**
- `db.complexity_snapshot` table (`file TEXT PRIMARY KEY, complexity INTEGER`); `save_complexity_snapshot` / `load_complexity_snapshot`.
- `current_complexity(root)` reuses `coach/hotspots._complexity` over the repo's source files.
- `find_regressions(root, conn)` diffs current vs the stored snapshot → emit `kind="regression"` recs for files whose complexity rose by ≥ a threshold (default +25% and +5 absolute) **or** are newly-present-and-complex. Gated behind `conn`-present.
- the snapshot is (re)written by **`torsor consolidate`** (the existing periodic maintenance pass) — establishing "what got worse since your last maintenance." Deterministic, no clock.

**Files:** new `coach/trend.py`, `db.py` (table + helpers), `coach/report.py`, `operations.consolidate` (snapshot update). **Risk:** low. **Tests:** snapshot a low-complexity file, raise its complexity, assert a `regression` rec; unchanged → none.

## Out of scope (documented, so we don't over-reach)
- Deep SAST / runtime vulnerability scanning (Veracode/Snyk/Semgrep) — torsor flags drift, it is not a security analyzer. (`forbid_pattern` already covers known dangerous idioms for those who want them.)
- Full secrets scanning (gitleaks/trufflehog own this).
- Skill atrophy / over-reliance — a human/education problem.
- Generating secure code or fixing vulns — the agent's job.

## Build order
P2 (impact — smallest, reuses edges) → P1 (deps — flagship, most surface) → P3 (coupling) → P6 (trend). Each: TDD (failing test → impl → green → commit), full suite green between tasks. After all four: an adversarial review, then README/CHANGELOG/version (→ 0.3.0)/dogfood-ADR updates, then merge to `main`.

## Testing strategy
- Every feature offline-testable on the default path: deps via a fabricated `.venv` top_level.txt; impact via a built index; coupling/trend via the `git_project` fixture.
- New deterministic helpers (degree, regression thresholds) unit-tested directly.
- No `SCHEMA_VERSION` bump needed unless a new table is added — `complexity_snapshot` (P6) is additive via `CREATE TABLE IF NOT EXISTS`; bump to 4 and rely on the existing additive-migration path.
- Keep the suite fully offline and green (currently 215).
