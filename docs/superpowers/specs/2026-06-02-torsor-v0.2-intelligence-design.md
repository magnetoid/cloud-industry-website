# torsor-helper v0.2 "Intelligence Release" — Design

**Goal:** Close the highest-value capability gaps found in deep competitive research (mem0/Zep/Graphiti, Aider/Serena/SCIP, CodeScene, ArchUnit/dependency-cruiser/ast-grep, Anthropic Contextual Retrieval, FlashRank, MMR, llms.txt/DeepWiki) — making torsor's *recall sharper*, its *map graph-aware*, its *guard CI-ready*, and its *coach a prioritizer* — while preserving every torsor invariant: local-first, Markdown is the source of truth, the SQLite index is rebuildable, the dependency surface stays small, and everything is deterministic + offline-testable.

**Provenance:** Each improvement below was generated by a research synthesis agent and then put through an *adversarial verification panel* that tried to kill it. Three candidates were killed and replaced with their sharpened, torsor-idiomatic alternatives (noted inline). The scope here is the surviving, sharpened set — 12 improvements in 4 themes.

---

## Non-negotiable invariants (apply to every task)

1. **No new required dependencies.** Optional features gate behind extras exactly like `fastembed` does. The default offline path (HashingEmbedder / no-git / no-vectors) must keep all tests deterministic.
2. **Markdown stays the source of truth.** Nothing here deletes user prose. New state is either in the rebuildable `.index/` DB or in clearly-marked frontmatter keys (`extra="allow"` already permits this).
3. **Determinism.** No `datetime.now()` or RNG inside ranking/search code paths (the Store's injected clock is the only time source, and it is not threaded into `hybrid_search`). Any decay must be a function of stored counters, not wall-clock.
4. **Graceful degradation.** Git absent → empty result. Vectors absent (keyword path / hashing path) → feature no-ops, never crashes.

## Theme A — Retrieval gets sharper

### A1. Contextual breadcrumb indexing  *(verified; scoped to snippet-safe fields)*
Prepend a structural breadcrumb to the text torsor *indexes* (not the text it *displays*). In `indexer.reindex`, build a breadcrumb from the fields the reindex loop already has per `Note`: tier name + posix path segments + title (e.g. `architecture > decisions > ADR 0002: Adapters depend on core`). Route it where it helps retrieval without polluting snippets:
- Prepend it to the **embedder input** (`embedder.embed([breadcrumb + "\n" + body])`).
- Write it into the **FTS `title` column** (`db.replace_fts(conn, path, breadcrumb + " " + title, body)`) — BM25 already weights the title column, and `db.body_of` never reads `title` for snippets.

The stored FTS `body` and the displayed snippet stay **byte-identical**. (Inspired by Anthropic Contextual Retrieval, ~49% fewer retrieval failures.) Drop the originally-proposed "parent ADR / nearest H2" breadcrumb parts — that data isn't on a `Note` and would make this not a pure-string change.

**Files:** `indexer.py` (+ a `_breadcrumb(note)` helper). **Risk:** low. A regression test asserts recall snippets are byte-identical before/after for a fixed corpus.

### A2. Section-aware snippet selection  *(replaces the killed cross-encoder reranker)*
The adversarial panel killed the FlashRank/ONNX reranker: its input is a 240-char keyword window, so a cross-encoder can't help and adds a heavy binary dep. The real lever is **snippet quality**. Improve `_snippet` (in both `search.py` and `recall.py`) to score candidate windows rather than taking the first keyword hit: split the body into paragraphs/sections (on blank lines / H2 headings), score each by summed term frequency (with a small bonus for a heading match), and return the best-scoring window trimmed to width. Pure Python, deterministic, zero deps.

**Files:** `search.py`, `recall.py` (share one helper, e.g. lift to a small `snippets.py` to keep both paths identical). **Risk:** low. Tests assert the snippet contains the densest term cluster, and that a single-term query still works.

### A3. Per-note importance multiplier  *(verified; scoped to access_count only)*
Notes never decay today (only coach recs do), so stale episodic noise competes forever with durable insight. Add a multiplier to `hybrid_search`'s final score (right where `_TIER_WEIGHTS` is applied, `search.py:65`) that floats frequently-recalled notes up while never burying fresh durable notes:

```
factor = floor[tier] + (1 - floor[tier]) * (1 - 1 / (1 + log1p(access_count)))
```

with `floor = 1.0` for CHARTER/ARCHITECTURE (never decay), and `~0.7` for EPISODIC (and a graded floor for MAP/ACTIVE). `access_count=0` maps to the floor (no cold-start suppression). Reuses the existing `access_count` column (durable across reindex via the `ON CONFLICT` clause). **No age term** — the panel showed episodic notes never stamp `updated`, and age would need a clock, breaking determinism. **No schema change.** Floors live in config (`IndexConfig`).

**Files:** `search.py`, `config.py`. **Risk:** low.

### A4. MMR diversification + tier-first packing + omitted marker  *(verified; budgeting already exists, so scope to the new parts)*
`max_tokens` budgeting already ships — don't re-add it. Add the genuinely-new parts inside `hybrid_search`'s packing loop:
- **MMR re-ranking** over candidate hits' stored `db.vectors`: `mmr = λ·rel − (1−λ)·max_sim_to_selected` (λ from config, default 0.7), **guarded to no-op when fewer than 2 hits have vectors** (so keyword/hashing paths are untouched). Drops near-duplicate journal/insight notes.
- **Tier-stable packing**: within the budget loop, prefer stable tiers (CHARTER→EPISODIC) on ties so scarce budget buys durable intent first.
- **Omitted marker**: when hits are dropped for budget, append a single sentinel `RecallHit` (e.g. `title="… N more omitted (budget)"`, `tier=EPISODIC`, empty snippet) so the agent knows recall was truncated.

**Files:** `search.py`, `config.py` (`mmr_lambda`). **Risk:** low. Tests assert determinism, no-op on <2 vectors, and the marker appears only on truncation.

## Theme B — The map gets real edges

### B1. Symbol dependency edges + honest ref counts  *(verified; scoped — NOT a call graph)*
Replace `cartographer.scan_repo`'s `blob.count(base)` substring heuristic (counts comments, strings, partial names) with a real AST reference pass. Record edges `(caller_symbol, referenced_name, role, resolved_module|NULL)` where `role ∈ {read, call, write}`, resolving **only the two cheap, reliable cases**: (a) same-module top-level defs, and (b) `from x import y` aliases mapped to their source module. Everything else stays `NULL` (degrade gracefully — the panel measured ~93% of call sites need import-alias + type inference, which is the fragile work we are NOT doing). **First payoff:** derive an accurate ref count from resolved edges to replace the substring count feeding `Symbol.refs` → `search_symbols` ordering, `render_map` ranking, and the coach reuse score. Skip per-attribute write tracking (`self.x` — unresolvable owner, noise).

**Files:** `cartographer.py` (new `extract_edges` / extended scan), `db.py` (new `symbol_edges` table + `replace_all_edges` + a `who_references(module, name)` query; bump `SCHEMA_VERSION` to 3), `models.py` (`SymbolEdge`). **Risk:** med (name resolution is the trap — keep it conservative and well-tested). Edges are additive and fully rebuildable.

### B2. Repo-fingerprint skip for `map_repo`  *(replaces the killed per-file incremental catalog)*
The panel killed per-file incrementalism: `refs` is a *global cross-file* count, so file-granular diffing serves silently-stale rankings. Ship the cheap, correct version instead: before scanning, compute a single repo-level fingerprint = hash of the sorted `(relpath, mtime, size)` of all non-ignored `*.py` files, stored in a `meta` row. If unchanged since the last `map_repo`, **skip the entire scan+render+reindex** (all-or-nothing preserves global-`refs` semantics exactly). A `--force` flag and a missing fingerprint both force a full run. Report `{"skipped": True}` when skipped.

**Files:** `operations.py` (`map_repo`), `cartographer.py` (a `repo_fingerprint(root)` helper), `cli.py`/`server.py` (`--force` / surfacing skip). **Risk:** low.

### B3. `torsor export` → llms.txt + Mermaid diagram  *(verified; depends on B1 for the diagram)*
A new `torsor export` command (and an opt emit on `map`/`consolidate`) serializing the pyramid to portable, human-diffable artifacts so torsor's memory isn't locked behind MCP:
- `.torsor/llms.txt` — the [llms.txt spec](https://llmstxt.org/) shape: H1 charter title, blockquote summary, then H2 sections per tier with annotated links to the key notes.
- A **Mermaid `graph TD`** of module-level import/reference edges (collapsed from B1's resolved edges) written into the T2 repo-map overview note — GitHub renders it inline.

Pure serialization of data torsor already owns; zero new deps, deterministic, offline. (Inspired by llms.txt, DeepWiki, Context7.)

**Files:** new `export.py`, `cli.py` (`export`), `server.py` (`export` tool), `paths.py` (`llms_txt`). **Risk:** low.

## Theme C — Guardrails grow up (CI-ready)

### C1. `require_import` + layering rule kinds  *(verified; ship require_import first, scope-as-from layering)*
`forbid_import`/`forbid_pattern` are both *prohibitive single-target* checks. Add two ADR rule kinds via the existing `_CHECKERS` registry:
- **`require_import`** (higher value, lower ambiguity — ship first): assert a mandated seam EXISTS. Inverts the violation model — emit exactly **one** file-level violation (`line=0`) when the import is *absent*, not per node. Files outside `scope` correctly yield zero violations.
- **`forbid_layer_import`**: "files matching `scope` may not import modules matching regex `target`" — reuse the existing `scope` as the "from" selector and `target` as a forbidden-import regex. **No new `from_path`/`to_path` Rule fields and no `$1` backreference machinery in v1** (add only if a concrete same-tier-sibling case demands it, with a malformed-group test).

**Files:** `guard.py` (`_require_import`, `_forbid_layer_import`, registry), `models.py` (doc the kinds). **Risk:** low (pure ast+re). Tests cover present/absent seams and out-of-scope no-ops.

### C2. Severity levels + `guard --json`  *(verified set, priority 11)*
Add an optional `severity ∈ {hint, info, warning, error}` (default `warning`) and a stable `rule_id` to `models.Rule`/`Violation`, and a machine-readable findings mode: `torsor guard --json` (CLI) + a `json` arg on the `check_drift` MCP tool, emitting `{rule_id, severity, file, line, message, source}`. `--strict` gains an optional `--severity` threshold so teams can "warn in dev, error in CI for these rules only" instead of all-or-nothing. **Files:** `models.py`, `guard.py`, `operations.py`, `cli.py`, `server.py`. **Risk:** low.

### C3. Drift baseline / ratchet  *(verified; scoped — committed config, tuple-count keyed, no fuzzy hash)*
`--strict` fails on ANY violation, so brownfield repos can never adopt enforcement. Add a baseline so `--strict` fails only on **new** drift:
- Store it at **`.torsor/baseline.json`** (a *committed, reviewable* sibling of the charter — **NOT** under `.index/`, which is disposable/rebuildable; a baseline is closer to source-of-truth config).
- Key strictly on `(file, rule_kind, target)` and **count violations per tuple** — `--strict` fails only when a tuple's current count *exceeds* its baseline count. This grandfathers existing debt without a fragile snippet-hash (which could let a genuinely-new violation silently inherit an old hash). Tolerant of line drift by construction.
- `torsor guard --update-baseline` is the **only** writer. `check_drift` stays side-effect-free (no auto-shrink during checks — shrink on the next explicit `--update-baseline`).
- Wire the baseline into the **server `check_drift` tool** too (it has no strict path today), not just the CLI.

**Files:** new `baseline.py`, `operations.py`, `cli.py`, `server.py`, `paths.py` (`baseline_file`). **Risk:** low.

## Theme D — The Coach prioritizes

### D1. Churn × complexity hotspot ranking  *(verified; scoped file-grained, transient)*
The coach flags hygiene but can't say *where to fix first*. Add `coach/hotspots.py`:
- Run `git log --name-only --pretty=format:` once (copy the `try/except OSError → []` degradation from `operations._git_changed`), count commits per `*.py` file.
- Compute one **per-file** complexity = decision nodes (`If/For/While/Try/BoolOp`) + LOC via a single `ast.walk` per file (file-grained, **no** per-function units, **no** new `Symbol`/`db` column — stays transient).
- Rank files by `churn × complexity`; emit the top 1–3 as `kind="hotspot"` Recommendations (`score=churn*complexity`, `key=f"hotspot:{module}"`, `severity="suggest"`), gated behind the `conn`-present branch in `report.assemble` **and** behind a `git rev-parse` check so non-git repos / offline tests cleanly yield zero hotspots. (Inspired by CodeScene, SonarQube.)

**Files:** new `coach/hotspots.py`, `coach/report.py`, `models.py` (doc the `hotspot` kind). **Test infra:** the first feature needing a git-populated fixture (a `git_project` fixture that inits a repo and makes commits). **Risk:** low.

### D2. ADR supersedes  *(replaces the killed write-time neighbor supersedes)*
The panel killed write-time neighbor supersession: torsor's journal/insights are aggregate files (not 1-note-per-file), so there's nothing to stamp. Scope to the one true 1-file-per-note surface — **ADRs**. Add `record_decision(..., supersedes=<adr-id-or-slug>)`: flip the named prior ADR's frontmatter to `status: superseded` and record `superseded_by: <new-id>` (both free under `extra="allow"`). Then `recall`/`get_intent` **down-weight or drop** `type == "decision"` notes whose `status == "superseded"` unless explicitly asked. Pure Markdown + the existing `status` field — no index lookup, no embeddings, no LLM, no schema change beyond a status filter.

**Files:** `operations.py` (`record_decision`, `get_intent`), `search.py` (status filter in the hits loop), `server.py` (`record_decision` gets a `supersedes` arg). **Risk:** low.

## Deferred (documented, not dropped silently)

- **Temporal-coupling edges from git co-occurrence** (synth #12, *not* in the verified top-10): medium risk, valuable, but better after B1/D1 land and a git fixture exists. Revisit in v0.3.
- **Per-session complexity-trend snapshots / "new findings only" coach view** (synth #13): depends on D1's complexity proxy; a history table is more state than this release should add. Revisit in v0.3.
- **Cross-encoder reranker** (synth #3): killed — superseded by A2 (fix the snippet first). Only reconsider if an A/B on the dogfooded corpus still shows top-8 misses after A2.
- **Per-file incremental symbol catalog** (synth #1): killed — unsound given global `refs`. Superseded by B2. Revisit only if `refs` is first redefined as a per-file local count.
- **Write-time neighbor supersedes** (synth #8): killed — superseded by D2 (ADR-scoped).

## Build order (dependency-sound)

A1 → A2 → A3 → A4  (retrieval; independent, foundational) →
B1 → B2 → B3  (B3's diagram needs B1's edges) →
C1 → C2 → C3  (C3 builds on the guard surface) →
D1 → D2  (D1 needs a git fixture; D2 is standalone).

Each task: TDD (failing test → implement → green → commit), full suite green between tasks. After all 12: an adversarial multi-dimension whole-branch review workflow, then an Opus whole-branch review, then README/CHANGELOG/version bump/dogfood-ADR updates, then merge to `main`.

## Testing strategy

- Every feature ships with unit tests on the default offline path (HashingEmbedder, no git, or no vectors) asserting determinism + graceful no-op.
- New fixtures: `git_project` (inits a repo + commits) for D1; a fixed multi-note corpus for A1/A2/A4 snippet/MMR assertions.
- Regression guard for A1: recall snippets byte-identical before/after.
- `SCHEMA_VERSION` bump (→3) for B1's `symbol_edges`; existing DBs upgrade additively (tables are `CREATE TABLE IF NOT EXISTS`), and the embedder-identity full-rebuild path already handles re-derivation.
- Target: keep the suite fully offline and green (currently 48 test files / 156 tests).
